# 跨境电商 AI Agent 共创计划 · 群成员必读

> 本文档是群组的入门手册 + 工作方法论，从「我们是谁」到「怎么把产品做出来」，全在这里。
> 新成员请从头读完，老成员有疑问随时来查。

---

## 我们是谁 & 为什么做这件事

30 人共创小组，目标是用 AI Agent 赋能 ToC 跨境电商卖家。
覆盖平台：亚马逊、TikTok Shop、Temu、独立站，以及一切有机会的电商场景。

### 对群成员自身的目标（二选一，必须达到其一）

| 目标 | 说明 |
|------|------|
| **求职方向** | 通过本项目的实战积累，找到 **年薪 50 万人民币** 的工作 |
| **创业方向** | 自己有店铺，AI Agent 落地后至少带来 **50 万元纯利润** |

> 达不到以上任一目标，此群可以解散。每一步都要认真做、做到位。

### 对客户（卖家）的目标

终极使命：**赋能 ToC 跨境电商卖家**，帮助他们降本增效、解决真实痛点、可量化地提高收入。

## 全局工作流程

```
第一步          第二步            第三步          第四步
痛点收集  →  评分 & 筛选  →  Agent 定义  →  Vibe Coding
（发现问题）  （选对问题）    （定义解法）    （做出来）

每步都有明确的「完成标准」，达不到不进入下一步。
约 6 周出第一个可推广的产品。
```

没有方法论的团队：收集 100 个痛点 → 不知道从哪下手 → 两周后还在争方向 → 做出来卖家不买单。
有方法论之后：用同一套格式描述痛点 → 数字打分定优先级 → 写代码前已知道「成功」长什么样。

## 第一步：痛点收集

### 核心原则：区分「症状」和「根因」

| 类型 | 示例 | 价值 |
|------|------|------|
| 症状（表象） | "上架太慢了" | 低 — 不知道慢在哪 |
| 根因（本质） | "同一款产品要在三个平台分别手写描述，关键词还不互通" | 高 — 知道怎么解决 |

**收集痛点时，一定要追到根因。**

### 痛点卡片格式

在 `痛点_问题_需求/职能痛点矩阵.md` 对应职能模块下，用以下格式填写：

```
**[痛点标题]**
- 场景：什么情况下会遇到这个问题？
- 当前做法：现在怎么解决（或忍受）的？
- 问题所在：具体卡在哪里？
- 损失量化：大概每周/每月损失多少时间或金钱？
- 期望结果：理想状态是什么样的？
- 平台：[亚马逊] / [TK] / [Temu] / [独立站] / [通用]
- 提交人：（昵称）
```

**好的示例：**
```
**客服时区问题导致响应超时**
- 场景：买家在美国白天（北京时间凌晨）发来咨询，要求 24 小时内回复
- 当前做法：要么熬夜盯消息，要么用平台自动回复（买家体验差）
- 问题所在：没有能理解上下文、自动给出专业回复的工具
- 损失量化：每月因响应超时扣分 3-5 次，影响排名；雇客服月薪 5000+
- 期望结果：AI 能在我睡觉时自动回复 80% 的常规咨询，异常的推送给我
- 平台：[通用]
- 提交人：张三
```

**差的示例（不要这样写）：**
```
- 客服太麻烦了 [通用]（张三）
```

### 第一步完成标准

- 每个职能模块至少 **5 条完整痛点卡片**
- 所有痛点都有**损失量化**（估算即可）
- 没有"运营太难了"这种无法拆解的描述

## 第二步：痛点评分 & 筛选

### 四维评分（管理员 + 2 名成员共同打分取平均）

| 维度 | 1 分 | 5 分 |
|------|------|------|
| **严重程度** | 轻微不适 | 不解决会亏损/封号 |
| **发生频次** | 偶尔 | 每天都有 |
| **AI 可行性** | 几乎无法解决 | 完全可以解决 |
| **市场规模** | 极少数卖家 | 绝大多数卖家 |

### 优先级公式

```
优先级得分 = 严重程度 × 频次 × AI可行性 × 市场规模（最高 625 分）

≥ 300 分 → 进入候选方向池
≥ 200 分 且市场上没有好工具 → 同样进入候选池
< 150 分 → 暂时搁置
```

### 最终过滤问题

> **「如果这个 Agent 存在，卖家愿意每月付多少钱？」**

- ≥ $50/月 → 商业价值够，可以做
- < $20/月 → 重新考虑，或找更高价值的切入点

### 第二步完成标准

- 所有痛点卡片完成评分
- 确定 **1-3 个核心方向**（不能更多，资源不够分散）
- 每个方向有明确的付费意愿预估

---

## 第三步：Agent 定义

> **在写第一行代码之前，把 Agent 定义清楚。这步做扎实，开发快 3 倍。**

在 `方向/` 目录下为每个方向创建文档：

```
# [Agent 名称]

## 解决的核心问题
（一句话，来自痛点根因）

## 目标用户
- 用户画像：做什么平台、什么规模、什么角色
- 典型场景：他在什么情况下用这个 Agent？

## Agent 做什么（功能边界）
### 做
- 功能 1
### 不做（明确排除）
- 排除项（原因）

## 输入 & 输出
| 输入 | 输出 |
|------|------|
| 用户提供什么 | Agent 返回什么 |

## 成功标准（可量化）
- 例：客服响应时间从 8 小时 → 30 分钟以内

## 竞品分析
| 竞品 | 优点 | 缺点 | 我们的差异化 |
|------|------|------|-------------|

## 技术可行性预判
- 需要哪些 API / 数据源？
- MVP 最简版本是什么？
```

### 第三步完成标准

- 定义文档通过全员审阅，没有明显逻辑漏洞
- 至少 **3 名真实卖家**确认「这个我会用」
- MVP 范围确定，2 周内可以出可演示版本

---

## 第四步：Vibe Coding

**先跑起来，再变好。** 不要在没有代码的情况下讨论架构。

### 开发节奏（每轮 1 周）

```
周一 → 确定本轮目标（一个具体功能）
周三 → 中期检查，遇到问题当天解决
周五 → 演示给群里，收集反馈
周末 → 整理文档，准备下轮
```

### 任务规则

- 每个任务 **一个人认领，两天内完成**
- 完成后在 `AI_agent/` 对应文档里标记，不要只在群里说
- 提交格式：`[模块名] 做了什么 / 修了什么`
  - 例：`[客服Agent] 实现亚马逊消息自动回复`

### 第四步完成标准（整个项目）

- 真实卖家用了一个月
- 至少有**一个可量化结果**（节省多少时间 / 增加多少收入）
- 卖家愿意推荐给同行

## 时间规划

| 步骤 | 时长 | 产出 |
|------|------|------|
| 痛点收集 | 1 周 | 完整痛点卡片库 |
| 评分 & 筛选 | 3 天 | 1-3 个核心方向 |
| Agent 定义 | 1 周 | Agent 定义文档 |
| MVP 开发 | 2 周 | 可演示版本 |
| 真实用户测试 | 2 周 | 量化结果 |
| **合计** | **约 6 周** | 第一个可推广的产品 |

## 常见错误 & 避免方法

| 错误 | 症状 | 避免方法 |
|------|------|---------|
| 功能蔓延 | "顺手再加一个功能" | 写进 Backlog，本轮不动 |
| 过度设计 | 还没写代码就在讨论架构 | 强制先跑通最简版本 |
| 痛点没验证 | 凭感觉选方向 | 必须有卖家愿意付费的证据 |
| 只看不做 | 群里很热闹，进度为零 | 每个任务必须有明确认领人 |
| 完美主义 | MVP 做了 6 周还没演示 | MVP = 跑通核心流程就算完成 |
